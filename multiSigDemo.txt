──────────────────────────────────────────────────────────────
CONSENSUS WORKFLOW — HOW RECORDS ARE AGREED UPON AND COMMITTED
──────────────────────────────────────────────────────────────

Overview
  Before a record can be queried using multi-signature verification, it must
  first be committed to the distributed ledger through a consensus process.
  Consensus ensures that a record is genuine and agreed upon by the majority
  of inventory nodes before it is permanently stored.

  The rule is simple: a record must exist in at least 3 out of 4 inventory
  nodes before it is considered valid and committed to the ledger.


─── Function: canonicalRecordKey(record) ────────────────────

  Purpose:
    Converts a record into a single string that uniquely represents its content.
    This string is used as a key to group and compare records across nodes.

  How it works:
    Joins the four record fields with a | separator:
      ItemID | ItemQty | ItemPrice | location

  Example:
    { ItemID: "1", ItemQty: "1", ItemPrice: "1", location: "B" }
    → "1|1|1|B"

  Why it matters:
    Two records from different nodes are considered identical only if they
    produce the exact same canonical key. This prevents partial matches
    (e.g. same ItemID but different quantity) from being counted as agreement.


─── Function: buildInventoryEntry(record) ───────────────────

  Purpose:
    Wraps a raw record into the structured format stored in inventoryX.json.

  How it works:
    Reads originNode from the record and produces:
      {
        original    : "B"           ← which node originally submitted this
        record      : { ...record } ← the full record data
        recordString: "1|1|1|B"    ← the canonical key for quick reference
      }

  Why it matters:
    Every committed entry on the ledger carries the canonical key alongside
    the record so future lookups can compare efficiently without re-computing.


─── Function: readJsonFile(filePath) ────────────────────────

  Purpose:
    Safely reads a JSON file from disk and returns its contents as an array.
    Returns an empty array if the file does not exist or is empty.

  Why it matters:
    Inventory and pending files may not exist yet (e.g. on first run).
    This function prevents the system from crashing in those cases.


─── Function: writeJsonFile(filePath, data) ─────────────────

  Purpose:
    Saves an array of records back to a JSON file on disk in a readable format.

  Why it matters:
    Used by both the commit step (writing to inventoryX.json) and the cleanup
    step (rewriting pendingX.json after removing committed records).


─── Function: buildCandidates(pendingByInventory) ───────────

  Purpose:
    This is the core of the consensus check. It reads all pending records
    from all 4 nodes and counts how many nodes hold each unique record.

  How it works:
    1. Loops through every node's pending list.
    2. For each entry, computes its canonical key.
    3. Groups entries by canonical key using a map.
    4. Tracks which nodes (sources) hold each unique record.
    5. Returns a list of candidates with their count and source list.

  Example:
    pendingA: [ "1|1|1|B" ]
    pendingB: [ "1|1|1|B" ]
    pendingC: [ "1|1|1|B" ]
    pendingD: [ "1|1|1|B" ]

    → candidates: [
        { record: {...}, recordKey: "1|1|1|B", count: 4, sources: [A,B,C,D] }
      ]

    Any candidate with count >= 3 passes consensus.

  Why it matters:
    This function determines whether the network has reached agreement.
    A record held by only 1 or 2 nodes is not trusted — it could be
    a tampered or isolated copy. Requiring 3 out of 4 means even if one
    node is compromised, the honest majority still governs the result.


─── Function: readAllPending() ──────────────────────────────

  Purpose:
    Reads all four pendingX.json files at once and returns them together.

  Returns:
    { A: [...], B: [...], C: [...], D: [...] }

  Why it matters:
    Provides a single consistent snapshot of all pending records across
    the network for the compare and commit stages to work from.


─── Function: commitConsensus(passedRecords) ────────────────

  Purpose:
    Permanently writes consensus-approved records to the ledger and
    removes them from the pending queues.

  How it works — Part 1 (write to ledger):
    For each passed record and each inventory node:
      1. Read the current inventoryX.json.
      2. Check if the exact same record (same canonical key) already exists.
      3. If not already there, append the new entry.
      4. Write the updated list back to inventoryX.json.

    Old entries are never deleted. New entries are always appended.
    This is blockchain behaviour — the ledger only grows, it never shrinks.

  How it works — Part 2 (clear from pending):
    For each node:
      1. Read the current pendingX.json.
      2. Remove any entries whose canonical key matches a committed record.
      3. Write the remaining entries back to pendingX.json.

  Why it matters:
    This is the point of no return. Once committed, a record is part of the
    permanent ledger on all 4 nodes. Clearing the pending queue ensures the
    same record is not processed again in future consensus rounds.


─── Function: processConsensus(stage) ───────────────────────

  Purpose:
    The main entry point for the consensus workflow. Controls which stage
    runs based on the stage parameter passed from the web interface.

  Stage: 'compare'
    - Reads all pending files via readAllPending().
    - Runs buildCandidates() to group and count records.
    - Filters candidates to those with count >= 3.
    - Returns the result for display — nothing is written yet.
    - If at least one candidate passes, the Commit button is shown.

  Stage: 'commit'
    - Repeats the compare step to get the current passed records.
    - If no candidates pass, returns a failure message.
    - If candidates pass, calls commitConsensus() to write to the ledger.
    - Returns a success message listing what was committed.

  Why the two-stage design:
    Separating compare from commit lets the user see what will be committed
    before it actually happens. This makes the consensus process transparent
    and auditable — important properties in a distributed ledger system.


─── Consensus Workflow Summary ──────────────────────────────

  Submit stage (handled by submitController):
    1. Origin node signs the record with its RSA private key.
    2. Record + signature are broadcast to all 4 nodes.
    3. Each node verifies the signature before storing in pendingX.json.
    4. Only verified records enter the pending queue.

  Compare stage:
    1. All four pending queues are read.
    2. Records are grouped by canonical key.
    3. Any record held by >= 3 nodes is a consensus candidate.
    4. Result is displayed to the user. No writes occur.

  Commit stage:
    1. Consensus candidates are identified again.
    2. Each passed record is appended to all four inventoryX.json files.
    3. Committed records are removed from all four pendingX.json files.
    4. The ledger is now updated across the entire network.

  After commit, the record is available for multi-signature query
  verification as demonstrated in Steps 2–6 above.


──────────────────────────────────────────────────────────────
HOW THIS IMPROVES TRUST, ACCOUNTABILITY, AND CONFIDENTIALITY
──────────────────────────────────────────────────────────────

Trust
  No single inventory node can return a result on its own. The multi-signature
  requires at least 3 out of 4 nodes to independently sign the same record.
  A corrupted or compromised node cannot forge the combined signature alone,
  so the querying user can trust that the result reflects the majority state
  of the distributed ledger.

Accountability
  Every partial signature is tied to a PKG-issued secret key that is derived
  from the node's identity string (e.g. "InventoryA:126"). This means each
  node's contribution to the combined signature is traceable back to that
  specific identity. If a node signs a fraudulent record, its identity is
  embedded in the signature and can be audited.

Confidentiality
  After the result is approved, the record hash is encrypted with PROCUREMENT's
  public key before transmission. Anyone intercepting the response sees only
  the ciphertext (protectedSeal) and the plaintext record, but cannot confirm
  whether the record matches the approved hash without PROCUREMENT's private key.
  Only the intended recipient can decrypt the seal and verify the integrity
  of the data they received.
